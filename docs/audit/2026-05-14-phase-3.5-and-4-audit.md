# Codebase audit — 2026-05-14 Phase 3.5 + Phase 4 wave

**Scope.** All 21 commits landed on `main` 2026-05-14 across two waves
(Phase 3.5 — `b1a39ec..c77372a`; Phase 4 — `f15de27..95f6df0`). Audited
by parallel `security-reviewer` and `code-reviewer` subagents plus direct
schema verification.

**Outcome.** **29 findings.** 5 CRITICAL, 9 HIGH, 11 MEDIUM, 4 LOW.

**Audit method.** Two adversarial passes — one focused on auth/crypto/
destructive-write surfaces (the new sync stack and cloud-bridge changes),
the other on logic gaps + missing call sites + edge cases. Findings
verified by direct grep / schema inspection where the agent's claim was
load-bearing.

**How to use this doc.** Treat the CRITICAL list as ship-blockers for any
production multi-host deployment. HIGH list as the next wave. MEDIUM /
LOW are documented for the operator's runbook but don't need urgent
fixes.

## CRITICAL — 5

### C1. `FAMILY_1_TABLES` lists 7 table names that DO NOT EXIST in the schema

**Location.** `lib/syncApply.ts:49-69`.

**Verified via.** `grep -q "CREATE TABLE IF NOT EXISTS <name>" lib/db.ts`.

| in tasks.yaml manifest | actual schema name |
|---|---|
| `temp_log_entries` | `temp_log` |
| `sanitizer_log` | `sanitizer_checks` |
| `sick_worker_log` | `sick_worker_reports` |
| `calibrations_log` | `thermometer_calibrations` |
| `pest_log` | `pest_control_log` |
| `sds_log` | `sds_registry` |
| `tphc_log` | `tphc_entries` |

**Impact.** **7 of the 21 family-1 HACCP tables silently skip
replication.** A peer pushing a `temp_log` insert → `familyOf('temp_log')`
returns `'unknown'` → applier returns `skipped-unknown-table` → the
operator dashboard counts it as "no problem." Regulatory hole the
moment the second site is live.

**Also affected in family 2/3:** `settlement_summaries` (family 2 — no
such table; settlements derived from `show_deals` + computed),
`recipes` (family 3 — no such table; JSON cache only).

**Fix.** Replace the bad names in `FAMILY_1_TABLES` and
`FAMILY_2_TABLES` / `FAMILY_3_TABLES` to match the live schema. Add
a startup-time assertion: `for (const t of FAMILY_*_TABLES) { if
(!schemaHasTable(t)) throw new Error(...) }` so a future schema rename
fails loud, not silent. Add a test that opens an in-memory DB and
verifies every family-set name appears in `sqlite_master.name`.
**Effort: S.**

### C2. `appendOp` is NEVER called from any source-table mutation

**Location.** `lib/syncFeed.ts:92` — only callers in the codebase are
test files.

**Verified via.** Greppable absence of `appendOp(` outside `tests/js/`.

**Impact.** **The producer side of the sync stack is wired but no real
data flows through it.** `sync_feed` will stay empty in production.
Every `replaySince` window is empty; the scheduler logs `no-new-ops`
every tick forever. T7a/T7b/T7c/Phase-4 #17 ship a complete sync
pipeline that's a no-op end-to-end.

**Fix.** For each Family-1 table mutation site, add `appendOp(...)`
inside the existing `db.transaction(...)` block — mirror the
`postAuditEvent` pattern. Required identity (sourceHost +
sourceStartedAt) needs a thin helper that reads from `lib/hubFailover.ts`
at boot — see C5 below for the bootstrap.

Start with the regulated paths (`app/api/checks/route.js` POST,
`app/api/cooling/route.js`, `app/api/temp-log/route.js`,
`app/api/receiving/route.js`, etc.). Each call site gets one extra
line inside the existing tx. **Effort: M** (15+ routes, but mechanical
once the helper exists).

### C3. Family-2 envelope with empty `where` wipes the entire table

**Location.** `lib/syncApply.ts:229-245`.

**Repro.** A trusted peer sends an op with
`rowJson = '{"where": {}, "rows": []}'` targeting `vendor_prices`.
The applier:
1. parses `where` as an object (passes `isObject` check)
2. `whereCols = Object.keys({}) = []` (passes unknown-cols check)
3. builds `DELETE FROM vendor_prices` (no WHERE)
4. INSERTs nothing — table is now empty.

**Impact.** Any compromised or buggy peer can wipe a receiver's
financial tables in one op. Especially dangerous because (a) it's a
single envelope, (b) the receiver's checkpoint advances after the
"apply" so re-fetch won't recover it, (c) only `vendor_prices_history`
snapshots survive on the originating host — receivers have nothing.

**Fix.** In `applyFamily2`:
```ts
if (whereCols.length === 0) {
  return { outcome: 'skipped-bad-payload', reason: 'empty where would wipe entire table' };
}
// AND/OR require location_id in where:
if (!whereCols.includes('location_id')) {
  return { outcome: 'skipped-bad-payload', reason: 'where must include location_id' };
}
```
Also: cap deleted-row count per envelope. If the DELETE would remove
> N (e.g., 10k) rows AND `rows.length` is small, log a
`sync_apply.suspicious_delete` audit row and refuse. **Effort: S.**

### C4. Family-1 INSERT OR IGNORE silently drops rows from cross-source PK collisions

**Location.** `lib/syncApply.ts:182-200`.

**Repro.** Two Lariat instances each maintain their own
`line_check_entries.id` AUTOINCREMENT. Instance A inserts `id=47`
("RTE salad, pass"). Instance B inserts `id=47`
("oil temp, fail"). When A replicates to B, the receiver's
`INSERT OR IGNORE INTO line_check_entries(...)` finds `id=47` already
exists → **silently no-ops**. A's row never lands on B.

**Impact.** "INSERT OR IGNORE on op_id-equivalent" is the documented
contract — but op_id lives on `sync_feed`, not on the regulated source
tables. The receiver's UNIQUE index is on `id`, which is per-host
AUTOINCREMENT. Cross-host collisions on small numeric ids are
overwhelmingly likely (every host starts from 1). The applier's
`applied` counter still increments because `db.run()` succeeds with
`changes=0` — no observability.

**Fix.** Pick one strategy and ship a migration:

**Option A (recommended)** — strip `id` from `rowJson` before insert,
let receiver assign its own. Requires `audit_events.entity_id` lookups
to traverse via `op_id` (already in `sync_feed`) rather than naked
numeric id. Bigger refactor but correct semantically.

**Option B** — add a `(source_host, source_id, source_started_at)`
composite unique index on each family-1 table via additive migration.
INSERT OR IGNORE on *that* tuple. Keep local id assignment, accept
that the same logical event has different numeric ids on different
hosts.

**Option C (interim mitigation)** — surface `changes=0` as a
`sync_apply.collision` audit row so at least the loss is observable.
Doesn't fix the bug; makes it visible.

**Effort: M (Option C) — L (Option A or B).** Blocks multi-host
shipping until at least Option C is in.

### C5. `LARIAT_CLOUD_BRIDGE_SECRET` written to settings.json at default file mode (644)

**Location.** `desktop/settings.ts:saveSettings` — `fs.writeFileSync(tmp, ...)`
has no `{mode}`, `fs.renameSync` inherits source mode.

**Comparison.** `lib/peerKeypair.ts:135` correctly does
`{mode: 0o600}` + belt-and-suspenders `chmodSync` for the Ed25519
private key. The HMAC secret for cloud bridge — equivalent sensitivity —
has no protection.

**Impact.** Any local user on a multi-user macOS install (or anyone
with backup access — Time Machine, iCloud Desktop) gets the secret.
Compromised secret lets an attacker push arbitrary payloads to the
cloud peer with valid HMAC signatures.

**Fix.** One-line change:
```ts
fs.writeFileSync(tmp, JSON.stringify(validated, null, 2), { encoding: 'utf8', mode: 0o600 });
// after rename:
fs.chmodSync(filePath, 0o600);
```
Plus: audit the Electron main process for any `console.log(settings)`
that would leak secret + path together. **Effort: S.**

## HIGH — 9

### H1. Signature replay window is ±60s = 120s total — no nonce cache

**Location.** `lib/peerTrust.ts:27, 144-150`.

**Impact.** A captured signed GET is replayable for up to **120 s** by
any LAN observer. Today the only damage is read amplification (sync
window is idempotent + read-only), but it's a free full-feed dump for
any passive sniffer. The 60s window symmetric around the timestamp is
larger than the doc comment implies.

**Fix.** Add a server-side seen-nonce cache keyed on `(pubkey, timestamp)`,
TTL = window. Any replay within the window hits the cache and is
rejected. Standard request-replay defense.

Alternatively, tighten to ±15 s and document the trade-off (more
clock-skew sensitivity, less replay window).
**Effort: S.**

### H2. PRAGMA table_info cache never invalidates across runtime schema changes

**Location.** `lib/syncApply.ts:117-131`.

**Impact.** `initSchema` runs `CREATE TABLE IF NOT EXISTS` and
`ALTER TABLE ... ADD COLUMN` on every boot — but in HMR or test
harnesses, the same process can re-run `initSchema` after the columns
cache has populated. A column added by a later migration silently
drops on every subsequent INSERT until process restart.

**Fix.** Export `bumpSchemaCache()` (or rename `_clearSchemaCacheForTest`
to `clearSchemaCache` and call it from `initSchema` after any
migration). At minimum: cache only on non-empty PRAGMA AND only when
the column count is non-zero. **Effort: S.**

### H3. Checkpoint advance uses `fromOp + ops.length` — infinite re-fetch loop on rowid gaps

**Location.** `lib/syncScheduler.ts:154-166`.

**Repro.** Server has `sync_feed.id` = 1, 2, 5 (3 + 4 from rolled-back
txs). Page returns rows 1, 2, 5 with `next_op: null`. Client sets
checkpoint to `0 + 3 = 3`. Next tick fetches `from_op=3` → server
returns row 5 again. Client sets checkpoint to `3 + 1 = 4`. Row 5
is **always re-fetched and re-applied** (op_id idempotency absorbs
the double-write, but the checkpoint never advances past 4).

**Fix.** Add a `last_seen_id` (or `high_watermark`) field to the
`/api/peers/sync-since` response. Always = the highest rowid the
server's SELECT observed, regardless of whether more rows exist.
Client checkpoints to `Math.max(currentCheckpoint, response.last_seen_id)`.
Don't synthesize from `fromOp + ops.length`. **Effort: S** (server
change + client change + 1 test).

### H4. 401 vs 400 oracle leaks auth state; `touchPeerLastSeen` fires before validation

**Location.** `app/api/peers/sync-since/route.js:60-82`.

**Issues:**
- A probe can distinguish "auth failed" (401) from "auth passed but
  params bad" (400) — confirms to an attacker that their signed-request
  setup is correct.
- `touchPeerLastSeen` fires **after** auth but **before** `peer_id` /
  `from_op` validation. A valid signature with bogus query params
  still bumps `last_seen_at` — useless audit trail entry, and lets a
  malicious-but-trusted peer keep `last_seen_at` fresh without
  actually fetching anything.

**Fix.** Validate `peer_id` / `from_op` shape **before**
`authenticateSyncRequest`. Return the same 401 + generic body for all
malformed-request cases until auth succeeds. Move
`touchPeerLastSeen` to fire only after `replaySince` returns
successfully. **Effort: S.**

### H5. Family-2 schema drift on receiver's column set fails open — widens DELETEs

**Location.** `lib/syncApply.ts:229-236`.

**Repro.** Producer sends `where: {location_id: 'X'}` (no vendor
scope). Receiver has a `vendor` column on `vendor_prices`. Receiver's
DELETE has no vendor filter → wipes ALL vendors at location X. The
"drop unknown cols" policy is correct for INSERT (forward-compat) but
inverts for DELETE.

**Fix.** Per-table whitelist of acceptable `where` shapes. At minimum
require `where` to cover the table's natural key + `location_id`.
Refuse anything narrower. **Effort: S.**

### H6. Two drainers race on `releaseAllClaimedRows` → double-delivery window

**Location.** `lib/cloudBridgeDrainer.ts:gracefulStop` +
`lib/cloudBridgeQueue.ts:releaseAllClaimedRows`.

**Repro.** Standalone drainer (PID A) claims row 5, mid-`pushBatch`.
In-process drainer (PID B) boots, then SIGTERM triggers `gracefulStop`.
PID B's `releaseAllClaimedRows` resets `claimed_at = NULL` on row 5
even though PID A still has it in flight. Between B's release and
A's ack, the row is claim-eligible again — a third drainer (cron-
triggered restart) can claim and double-push.

**Fix.** Add `claim_owner TEXT` to `cloud_bridge_outbox` (additive
migration). Populate with per-process UUID at claim time.
`releaseAllClaimedRows` should only reset rows owned by *this*
process. **Effort: S** (migration + 5-line change).

### H7. Settlement PDF has no Content-Security-Policy header

**Location.** `app/api/shows/[id]/settlement/pdf/route.js`,
`lib/settlementPrint.ts:155-162`.

**Today.** All five interpolation points (`bandName`, `date`,
`locationId`, `computedAt`, `costsOffTop[].label`) flow through
`escapeHtml`. The inline `<style>` and `<script>` blocks are static.
**Safe today.**

**Fragile tomorrow.** A future contributor who interpolates user data
into the STYLE block (per-tenant theme color is the obvious one)
opens CSS-injection or script-injection. Add CSP as defense-in-depth.

**Fix.** Add to the response headers:
```js
'Content-Security-Policy': "default-src 'none'; style-src 'unsafe-inline'; script-src 'unsafe-inline'",
'X-Content-Type-Options': 'nosniff',
```
(`'unsafe-inline'` is required for the inline blocks. Moving the
auto-print script to a CSP-nonce'd block is the more robust
follow-up.) **Effort: S.**

### H8. `getTableColumns` doesn't cache the negative case → O(N) PRAGMAs per tick on misbehaving peer

**Location.** `lib/syncApply.ts:119-131`.

**Repro.** A peer floods ops for a table that doesn't exist locally.
Every op runs `PRAGMA table_info` → returns `[]` → returns `null` →
NOT cached. Next op repeats. PRAGMA isn't free under WAL — N ops/tick
becomes a cheap remote DoS amplifier.

**Fix.** Cache the empty result too:
```ts
if (!rows.length) {
  COLUMNS_CACHE.set(tableName, EMPTY_SET);
  return null;
}
```
And treat `size === 0` as the "unknown table" signal in callers.
**Effort: S.**

### H9. `DEFAULT_KEYPAIR_PATH` evaluated at module load — defeats `resolveDataDir()` lazy contract

**Location.** `lib/peerKeypair.ts:69`.

**Inconsistency.** `lib/dataDir.ts` deliberately reads env at every
call so tests can set `LARIAT_DATA_DIR` after import. But
`DEFAULT_KEYPAIR_PATH = join(resolveDataDir(), 'peer-keypair.json')` is
a top-level const — captures the value at module load. Tests that
import `peerKeypair.ts` and then set `LARIAT_DATA_DIR` see the wrong
keypair path.

**Fix.** Change to a function:
```ts
export function defaultKeypairPath(): string {
  return join(resolveDataDir(), 'peer-keypair.json');
}
```
Update `loadOrCreateKeypair(path = defaultKeypairPath())`. **Effort: S.**

## MEDIUM — 11

### M1. `LARIAT_SYNC_PEER_KEY` defaults to `process.pid` — PIDs reused across reboots

**Location.** `lib/syncSchedulerLifecycle.ts:143-144`.

**Impact.** Identity should be `(host, started_at)` per
`lib/hubFailover.ts::peerKey()`. PID identity collides across reboots
— remote receiver's `replay_checkpoints` for "peer 1234" is wrong
after a host reboot reuses PID 1234.

**Fix.** Default to `peerKey()` from `lib/hubFailover.ts`. Refuse to
boot if `peerKey()` is unavailable. **Effort: S.**

### M2. `parsePeersEnv` accepts `file://` and `localhost:*` — SSRF surface

**Location.** `lib/syncSchedulerLifecycle.ts:88-101`.

**Impact.** If `LARIAT_SYNC_PEERS` is influenced (compromised settings
file, env-var injection), scheduler can be pointed at
`http://169.254.169.254/...` (AWS metadata) or `file:///etc/passwd`.
Scheduler signs the request — attacker gets a signed exfil channel.

**Fix.** Parse `baseUrl` via `new URL()`. Reject `file:`/`data:`/etc.
Reject localhost + RFC1918 unless `LARIAT_SYNC_ALLOW_PRIVATE=1`.
**Effort: S.**

### M3. `addPeer` auto-unrevokes via `ON CONFLICT DO UPDATE SET revoked = 0`

**Location.** `lib/peerTrust.ts:46-53`.

**Impact.** Revoked peer can be re-trusted by anyone calling
`addPeer`. Today only the repo helpers call it; once a management UI
ships, revocation becomes a soft state.

**Fix.** Either remove `revoked = 0` from ON CONFLICT (require
explicit `unrevokePeer`) or require an `unrevoke: true` arg.
**Effort: S.**

### M4. `verifyProof` swallows all errors → no degraded-crypto signal

**Location.** `lib/peerKeypair.ts:186-207`.

**Impact.** OOM, FIPS-mode disabled, OpenSSL error all return
"untrusted peer." Every sync request 401s with no log signal in a
degraded state.

**Fix.** Differentiate `Error.code === 'ERR_OSSL_*'` (real failures —
log at warn) from input-shape errors (silent false). **Effort: S.**

### M5. `setReplayCheckpoint` accepts any string for `peer_id` — DoS / pollution surface

**Location.** `lib/syncFeed.ts:219-232`.

**Impact.** Long or pathological string in `peer_id` lands as a
permanent DB row. Internal-only today, but `runPeerCycle` passes
`peer.feedKey` from `LARIAT_SYNC_PEERS` JSON — an operator typo
becomes durable.

**Fix.** Clip `peer_id` to 256 chars, reject control characters. Same
for `feedScope`. **Effort: S.**

### M6. `lib/jobLock.ts` hard-codes `process.cwd()/data/locks` — refactor missed it

**Location.** `lib/jobLock.ts:76`.

**Impact.** Relocated install (`LARIAT_DATA_DIR=...`) has DB + JSON
cache on the external drive but lockfiles in `cwd/data/locks`.
Concurrent-run prevention still works (both processes share cwd) but
restart from a different cwd orphans locks.

**Fix.** Import `resolveDataDir`, replace with
`path.join(resolveDataDir(), 'locks')`. **Effort: S.**

### M7. `lib/auditLog.mjs` hard-codes `process.cwd()/data/audit` at module load

**Location.** `lib/auditLog.mjs:10`.

**Impact.** Same split-brain as M6. `syncApply.ts` calls
`logAuditAction` for family-3 skips — those land in the wrong dir on
a relocated install.

**Fix.** Resolve lazily at write time:
```js
const auditDir = path.join(resolveDataDir(), 'audit');
```
**Effort: S.**

### M8. No operator path to populate `peer_trust` — sync route is permanently 401 in any new deploy

**Location.** Missing. `lib/peerTrust.ts::addPeer` exists but no CLI,
no `/management` route, no first-run setup populates it.

**Impact.** Operator sets `LARIAT_SYNC_PEERS`, starts the scheduler,
every `fetchSyncSince` returns 401. There's no operationally
documented path to bootstrap trust between two new instances.

**Fix.** Ship one of:
- `scripts/trust-peer.mjs --pubkey=<hex> [--label=...]` CLI
- `/api/management/peers` PIN-gated POST
- mDNS-driven pairing wizard
At least the CLI is bounded (~30 lines + a test). **Effort: S.**

### M9. `sync_feed` retention is unbounded — no GC

**Location.** `lib/db.ts` (table is APPEND-ONLY), no purge script.

**Impact.** Three-tablet HACCP cadence writes hundreds of rows/day per
peer. After a year: ~hundreds of thousands of rows. SQLite handles
this fine, but query performance + WAL size grow without a safe GC
policy.

**Fix.** Document the GC contract: `sync_feed.id < MIN(peer_id's
last_op_rowid)` is safe to delete. Write a maintenance script
`scripts/gc-sync-feed.mjs` that computes the floor and DELETEs below
it (with a dry-run flag). Wire into `data/scheduled-jobs.json` as a
weekly job. **Effort: M.**

### M10. mDNS discovery not wired to scheduler — peers are env-only

**Location.** `lib/mdnsDiscovery.ts` exists; `lib/syncSchedulerLifecycle.ts`
reads peers only from `LARIAT_SYNC_PEERS`.

**Impact.** Newly-joined peer requires env edit + restart. The whole
point of mDNS is dynamic discovery.

**Fix.** Have `bootSyncScheduler` subscribe to mDNS events and call
`scheduler.setPeers([...])` (need to add a `setPeers` method to the
scheduler handle). The env-var override stays as a manual escape
hatch. **Effort: M.**

### M11. `lib/syncClient.fetchSyncSince` doesn't bound `res.json()` body size

**Location.** `lib/syncClient.ts:106-120`.

**Impact.** A trusted-but-malicious peer (or a misconfigured peer
returning HTML for some reason) can serve a 1 GB JSON body —
`res.json()` buffers it. Memory DoS on the puller.

**Fix.** Check `Content-Length` header, refuse > 10 MB. For chunked
responses, wrap the stream in a length-tracking transformer. **Effort: S.**

## LOW — 4

### L1. `runPeerCycle` apply-skipped advance also affected by H3 / C-LOGIC-3

**Location.** `lib/syncScheduler.ts:166`.

**Note.** Same `fromOp + ops.length` advance logic when the window is
all family-3 skips. Same fix as H3 (server returns `last_seen_id`)
covers both.

### L2. `cloudBridgeDrainer.ts:127` `stillInQueue` logic has a minor cosmetic miss

**Location.** `lib/cloudBridgeDrainer.ts:127-133`.

**Issue.** `const stillInQueue = queue.deadLetterDepth() === 0 ||
hasRowAlive(batch.id);` — the `||` is technically correct for the four
outcome cases but the boolean shape is confusing. `hasRowAlive(batch.id)`
alone is the right predicate; the dead-letter-depth check is redundant.

**Fix.** Replace with `hasRowAlive(batch.id)`. Cosmetic only. **Effort: trivial.**

### L3. Audit-log management actions from `applyWindow.reasons` may render attacker-controlled column names

**Location.** `lib/syncApply.ts:applyWindow` reason strings carry
producer-supplied column names from `dropped`.

**Impact.** When a future `/management/sync` UI renders these reasons,
the column names must be HTML-escaped (they're attacker-influenced via
the producer's `rowJson`).

**Fix.** Document the requirement now; revisit when the UI lands.
**Effort: deferred.**

### L4. `lib/cloudBridgeDrainer.gracefulStop` doesn't return how long it waited

**Location.** `lib/cloudBridgeDrainer.ts:204-227`.

**Impact.** Operator running `npm run cloud-bridge:drain` and hitting
SIGTERM doesn't see how long the in-flight tick took to settle. Just
"released N claims." For incident response a "waited 2400 ms" line
would be useful.

**Fix.** Return `{ released: number, awaitedMs: number }` and have
the script log it. **Effort: trivial.**
