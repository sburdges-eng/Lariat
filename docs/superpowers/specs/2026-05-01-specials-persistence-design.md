# Specials Persistence — Save, List, Edit, Export

Date: 2026-05-01
Files: `lib/db.ts`, `middleware.js`, `app/_components/navRegistry.js`, `app/specials/page.jsx`, `app/specials/saved/**`, `app/api/specials/saved/**`, `tests/js/test-specials-saved-*.mjs`

## Goal

Add a manager-curated persistence layer to the existing Specials sandbox. Today the sandbox is fully ephemeral — pantry, prompt, AI answer, and scratch notes all live in component state and vanish on reload. Operators want to save sessions they like, come back to them, refine notes, and export them as a workbook-pasteable artifact when a special is ready to enter the master recipe workbook.

The chat surface at `/specials` stays unchanged and stays open to line cooks. Saving, listing, editing, and exporting are PIN-gated under `/specials/saved/*` and `/api/specials/saved/*`.

Recipes in this codebase are owned by Excel workbooks and surfaced through `data/cache/recipes.json` (regenerated by ingest). This spec deliberately does **not** write to `entities_recipes` / `bom_lines` — exporting produces a CSV the chef pastes into the master workbook on the next ingest pass. That keeps the workbook-as-source-of-truth invariant intact.

## Schema

New table in `lib/db.ts::initSchema` (never edit existing DDL in place — add a fresh `db.exec` block per the schema-change rule):

```sql
CREATE TABLE IF NOT EXISTS specials (
  id                 TEXT PRIMARY KEY,             -- UUIDv7 via lib/entities
  location_id        TEXT NOT NULL DEFAULT 'default',
  name               TEXT NOT NULL,
  pantry_text        TEXT NOT NULL DEFAULT '',
  prompt_text        TEXT NOT NULL DEFAULT '',
  ai_answer          TEXT NOT NULL DEFAULT '',
  ai_model           TEXT NOT NULL DEFAULT '',
  cost_breakdown     TEXT,                         -- JSON, nullable; rows from computeSandboxCost
  cost_total         REAL,
  scratch_notes      TEXT NOT NULL DEFAULT '',
  sources            TEXT,                         -- JSON, nullable; grounding sources from buildGroundedContext
  last_exported_at   INTEGER,                      -- bumped on every successful export; null = never exported
  created_at         INTEGER NOT NULL,
  updated_at         INTEGER NOT NULL,
  archived_at        INTEGER                       -- soft-delete; null = active
);
CREATE INDEX IF NOT EXISTS idx_specials_loc_created ON specials(location_id, created_at DESC);
CREATE INDEX IF NOT EXISTS idx_specials_active ON specials(location_id, archived_at) WHERE archived_at IS NULL;
```

Captured session fields (`pantry_text`, `prompt_text`, `ai_answer`, `ai_model`, `cost_breakdown`, `cost_total`, `sources`) are immutable after insert. `name` and `scratch_notes` are the only mutable fields. This is enforced server-side, not just in the UI.

## API

All routes are PIN-gated. `middleware.js` adds `/specials/saved` (page) and `/api/specials/saved` (API tree) to its gated path list. Each route also re-checks `lib/pin.ts::hasPinCookie()` so curl/replay can't bypass the middleware.

| Method + path | Body | Response | Notes |
|---|---|---|---|
| `POST /api/specials/saved` | `{name, pantry_text, prompt_text, ai_answer, ai_model, cost_breakdown?, cost_total?, scratch_notes?, sources?}` | `{id}` | Generates UUIDv7. `cost_breakdown` and `sources`, if present, must be valid JSON strings or arrays/objects (server stringifies). |
| `GET /api/specials/saved?location=…` | — | `{items: [{id, name, created_at, cost_total, snippet, last_exported_at}]}` newest first; archived excluded | `snippet` is the first ~120 chars of `ai_answer`. |
| `GET /api/specials/saved/:id` | — | full record | 404 if not found OR if it belongs to a different location. |
| `PATCH /api/specials/saved/:id` | `{name?, scratch_notes?}` only | `{ok: true}` | Any other field in body → 400 listing rejected fields. Bumps `updated_at`. |
| `DELETE /api/specials/saved/:id` | — | `{ok: true}` | Sets `archived_at`. Idempotent (deleting an already-archived row 200s with no-op). |
| `POST /api/specials/saved/:id/export` | `{slug, yield_qty, yield_unit, category?, procedure_override?}` | `{recipe_row, ingredient_rows, skipped, csv}` | Updates `last_exported_at`. Re-export is allowed (just refreshes the timestamp). 410 if `archived_at` is set. |

Validation rules:

- `name`: 1–200 chars after trim.
- `slug`: matches `^[a-z0-9-]+$`, 1–80 chars, must not collide with an existing `entities_recipes.slug` for the location (read-only check; no write).
- `yield_qty`: `> 0` finite number.
- `yield_unit`: 1–32 chars after trim.
- `category` (optional): ≤ 64 chars.
- `cost_breakdown` (when posting): if present, must be a JSON array; rows are stringified for storage as-is.
- Disallowed PATCH fields: any key other than `name` and `scratch_notes` → 400 with the rejected key list.

## Export shape

Server reads the saved special, parses `cost_breakdown` JSON for the ingredient list, and produces a single CSV with two labeled sections. The same data is also returned as structured `recipe_row` and `ingredient_rows` so the detail page can render preview tables.

```
# RECIPE
slug,display_name,yield_qty,yield_unit,category,procedure
my-pork-belly-app,Pork Belly App,12,portions,appetizer,"Sear belly..."

# INGREDIENTS
ingredient,qty,unit,vendor_match,note
Pork Belly,2,lb,Sysco Pork Belly Skin-On,
Tomato (soft),0.5,case,,unmatched — pick a vendor item before paste
```

Mapping:

- `slug`, `yield_qty`, `yield_unit`, `category` come from the export request body.
- `display_name` is `specials.name`.
- `procedure` is `procedure_override` if provided, else `ai_answer` with any trailing cost-breakdown markdown blockquote stripped.
- Each ingredient row maps from a `cost_breakdown` row: `ingredient = item`, `qty = req_qty`, `unit = req_unit`, `vendor_match = match` (empty string if `cost === null`), `note = ''` for matched rows or `'unmatched — pick a vendor item before paste'` for `cost === null` rows.
- CSV escaping: double-quote any field containing comma, double-quote, or newline, and double the inner quotes (RFC 4180).
- `skipped` in the response is the array of unmatched ingredient rows so the UI can flag them visually.

## Data flow

Save:

```
client (page.jsx)
  → POST /api/specials/saved (PIN cookie required)
    → middleware.js verifies PIN cookie
    → route validates body
    → db.transaction:
        INSERT INTO specials (...)
        appendFileAudit('specials.create', {special_id, name, location_id})
    → return {id}
```

Edit / delete / export follow the same pattern with their respective audit actions: `specials.update`, `specials.delete`, `specials.export`.

`appendFileAudit` is `lib/auditLog.mjs` (existing helper). All four mutations share the same `db.transaction(...)` wrapper so an audit-write failure rolls back the row mutation.

No `audit_events` row is emitted — specials are not a regulated entity (HACCP / labor / financial). File audit is the right tier per `docs/PATTERNS.md §3`.

## Error handling

- Missing `name` → 400 `{error: "name required"}`.
- Empty captured session (`pantry_text` AND `prompt_text` AND `ai_answer` all empty) → 400 `{error: "no session content to save"}`. (Prevents accidental empty rows.)
- Invalid JSON in `cost_breakdown` or `sources` → 400 `{error: "invalid <field> JSON"}`.
- PATCH with disallowed fields → 400 `{error: "fields not editable", rejected: [...]}`.
- Cross-location id (id exists but other location's row) → 404 (don't leak existence).
- Export of archived special → 410 `{error: "special is archived"}`.
- Export with `yield_qty <= 0` or empty `yield_unit` → 400 with the failing field name.
- Export with slug collision against `entities_recipes.slug` (same location) → 409 `{error: "slug already exists", slug}`. Read-only check; no DB modification.

Export inputs (`slug`, `yield_qty`, `yield_unit`, `category`, `procedure_override`) are transient — they are not stored on the special row. Re-exporting requires re-entering them. Only `last_exported_at` persists. This keeps the special row a pure session-snapshot record and avoids a half-built recipe schema living in the specials table.
- 401 from PIN gate → standard middleware redirect for pages, JSON `{error: "unauthorized"}` for API.

## Out of scope

- Promotion that writes to `entities_recipes` or `bom_lines` (rejected — fights the workbook ownership model).
- Sidecar JSON merging promoted specials into `/recipes` (deferred; could be a separate spec).
- Versioning / iteration history for the same logical special.
- Search, filter, tags on the list view.
- Auto-save while chatting (explicit save only).
- A "promote and re-ingest" flow that round-trips into the master workbook automatically.
- Tracking who exported (no per-user identity exists; PIN cookie is location-level, not user-level).

## Risks / notes

- The current `/api/specials` POST returns `{answer, model, location_id, sources, latencyMs, disclaimer}`. The save flow needs `cost_breakdown` and `cost_total` in that response too. Two options for the implementer: (a) add them to the existing route's response, or (b) re-derive client-side from the answer markdown. (a) is cleaner and is what this spec assumes.
- `entities_recipes.slug` collision check is read-only but adds a DB read on every export. Acceptable — exports are rare.
- `cost_breakdown` JSON shape is the existing `computeSandboxCost` row shape (`{item, req_qty, req_unit, match, pack_size, pack_unit, pack_price, cost, note?}`). The export pipeline must tolerate older saved rows that pre-date any future shape changes — write the parser defensively (drop unknown fields, default missing fields to empty/null, never throw on a partial row).
- Soft-delete via `archived_at` means hard cleanup is a separate concern; the list query and indexes already exclude archived rows so this is fine indefinitely.
- File audit appends are not transactional with SQLite by default. `lib/auditLog.mjs` is a `fs.appendFileSync` — wrap the audit call inside the same `db.transaction(...)` so a synchronous file failure throws and rolls back the SQLite mutation (existing pattern in this codebase).

## Success criteria

- Schema migration is idempotent (`npm run test:schema` passes).
- A chef can chat at `/specials`, click "Save this special" with a name, and see it on `/specials/saved` with PIN.
- Edit name and scratch_notes; immutable session fields stay frozen.
- Soft-delete from the detail page; the row disappears from the list and a re-delete is a no-op.
- Export downloads a two-section CSV that round-trips into a workbook paste; unmatched ingredients are flagged.
- All file-audit rows are present in `data/audit/management-actions.jsonl`.
- All four test files pass under `node --test` / Jest.
- `gitnexus_detect_changes()` shows only the expected symbols affected.
