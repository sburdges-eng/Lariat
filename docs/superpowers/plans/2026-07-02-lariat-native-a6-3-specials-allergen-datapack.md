# LariatNative A6.3 — specials, allergen-lookup, datapack-search

Branch: `feat/lariat-native-a6-3-specials` · Base: `origin/main` (c0ab963)

Ports three web surfaces into LariatNative with layer-by-layer TDD
(Compute → Records → Repository → View/VM → A0 self-registration). The web
sources are the spec; native tests pin parity against the same oracles the
web tests use.

## Scope

| Board | Web spec | Native id (tier) | Kind |
|---|---|---|---|
| Saved specials | `app/specials/saved/*`, `app/api/specials/saved/*` (4 routes), `lib/specialsValidators.ts`, `lib/specialsExport.ts`, `lib/specialsPromotion.ts` | `manager.specials` (.manager) | PIN-gated read/write |
| Allergen lookup | `app/allergen-lookup/*`, `app/api/allergens/attestations`, `lib/allergenAttestations.ts` | `safety.allergenLookup` (.safety) | SAFETY-critical; audited attestation writes |
| Datapack search | `app/datapack-search/*`, `app/api/datapack/search`, `lib/datapackSearch.ts` | `cook.datapackSearch` (.cook) | Read-only search |

### Tier choices (documented per brief)

- **`manager.specials` → .manager** — `/specials/saved` is a middleware
  SENSITIVE_PREFIX on the web (manager PIN); the board manages menu R&D
  records and promotes onto the costing surface. Mac/manager side. No new
  `FeatureTier` case is introduced — the existing manager tier carries it,
  keeping the `FeatureCatalog.swift` edit minimal per the scope contract.
- **`safety.allergenLookup` → .safety** — the web treats it as a
  food-safety surface (cook-readable allergen answers + manager-attested
  house-recipe allergen lists). Fail-loud chip posture is preserved.
- **`cook.datapackSearch` → .cook** — open reference lookup for the line
  ("ingredient, brand, regulation"); no PIN anywhere on the web route.

## Phase-B boundary (specials)

`app/api/specials/route.js` (the LLM sandbox: Ollama chat, grounded
context, `cost_special` action extraction, sandbox costing) is **NOT
ported** — it is the kitchen-assistant / Phase B surface. This wave ports
only the SAVED-specials persistence + rules:

- create / list / get / patch / soft-delete (`/api/specials/saved`,
  `/api/specials/saved/[id]`)
- CSV export (`…/export`) — `specialsValidators` + `specialsExport`
- menu promotion (`…/promote`) — `specialsPromotion` end to end

Native has no sandbox, so `SpecialsView`'s "create" path is limited to the
fields the POST route accepts; new specials normally arrive via the web
sandbox. The native board is management of the saved corpus (rename,
notes, delete, export, promote) at full rule parity.

`lib/dealPoints.ts` was listed in the brief "as used" — it is **not
referenced by any specials surface** (it is the shows/settlement-math
module), so nothing of it is ported here.

## Audit posture (mirrored exactly, pinned by tests)

| Write | audit_events (in-tx) | JSONL (`logAuditAction`) |
|---|---|---|
| specials create/patch/delete/export | none on web → none native | `specials.create/update/delete/export` **inside** the same write transaction (web wraps both in `db.transaction`) |
| specials promote | `entity=specials_promotion`, action `insert`/`update`, payload `{special_id, menu_item_name, servings, component_count, skipped_count}` — in the SAME transaction as the `dish_components` + `specials_promotions` writes | `specials.promote` after commit (web logs at the route layer, post-txn) |
| allergen attestation | `entity=allergen_attestation`, action `insert`, `actor_cook_id=attested_by`, payload = full row — in the SAME transaction as the append-only INSERT | none |

- Rule failures throw typed `WriteError`s BEFORE any write (or the audit
  row), matching the web's 400/404/409/410 branches.
- `actor_source`: web stamps `pic_ui` (promote) / `manager_ui` (attest).
  Native stamps `native_mac` on both — both surfaces are PIN-gated
  (documented divergence, consistent with every prior native wave;
  asserted in tests).
- No `withIdempotency` port — native calls are direct in-process; the
  divergence is asserted in the repository tests per the A4/A5 precedent.
- Attestations are append-only: corrections are fresh rows, never
  UPDATE/DELETE (pinned by test).

## Deliberate deferrals / divergences

1. **Datapack semantic + hybrid search (`op=semantic`, `op=hybrid`) are
   NOT ported.** They require the BGE-small ONNX embedding model via
   transformers.js plus multi-GB `vectors.npy` buckets. Read-only search
   modes, not write paths — the native board ships lexical FTS5 (BM25),
   direct lookups, and stats, and labels the board accordingly. Porting
   would need a Swift ONNX/CoreML embedding runtime — flagged as a
   follow-up, not attempted here. (`prewarmDataPack` is likewise N/A.)
2. **Datapack availability** mirrors the web: data root =
   `<dataDir>/lariat-data` symlink, else `LARIAT_DATA_ROOT`; missing pack
   → `available() == false`, searches return `[]`, the view shows the
   web's exact unavailable copy. Both connections read-only +
   `PRAGMA query_only = ON`.
3. **URL-builder helpers** (`buildLookupUrl`, `offProductUrl`) exist on
   the web to route fetches; native calls the repository directly. The
   routing *decision* (GTIN → direct product lookup vs FTS search) is
   ported as `AllergenLookupHelpers.route(for:)` with every oracle case;
   the URL-string forms are web-transport-specific and not reproduced.
4. **`ai_answer` is intentionally uncapped** on create (web comment:
   clipping would corrupt LLM markdown) — mirrored.
5. Native list view joins promotions like the `/specials/saved` server
   page (badge "On menu as …"), which the API route alone does not.
