# Lariat Kitchen Assistant v3 — Implementation Plan

> Follow-on to the v2 build (`docs/superpowers/plans/2026-07-09-lariat-ka-v2-local-model.md`).
> v2 trained a working pipeline but the flipped model (Qwen3-8B) had a disqualifying
> UI JSON-leak and was rolled back to DeepSeek. A 6-axis research fleet
> (2026-07-10) found the real root causes — recorded here as the v3 fixes.

**Goal:** A local kitchen-assistant model that *provably* beats the DeepSeek-R1-14B
incumbent on a trustworthy gate, emits the action-JSON contract with structural
guarantees (no UI leak possible), and runs faster on a semi-in-use 16GB M4.

**Owner decision:** Full v3 rebuild (all 6 workstreams). Budget uncapped;
cloud cost is small (<~$100), engineering time is the real cost.

## Root causes found (why v2 was mediocre)

1. **Loss-masking bug** (`train.py`): `SFTConfig` has no `assistant_only_loss` and
   the custom CHATML has no `{% generation %}` markers, so loss is computed over the
   full ~18k-token sequence. >95% of gradient went to reconstructing the fixed
   prompt, under-training the ~200-token assistant output where JSON-discipline
   lives — and making `val_loss` a flat ~0.005 prompt-copying artifact, useless for
   selection.
2. **Train/serve template mismatch** (`train.py`): the empty `<think>` block only
   lives in the `add_generation_prompt` branch, which full-conversation SFT never
   triggers → trained without it, served with it. (The v2 commit claiming this was
   "wired into training data" was wrong.)
3. **Wrong base**: there is no Qwen3-8B-Instruct-2507; picking Qwen3-8B forced the
   hybrid-thinking model. `Qwen3-4B-Instruct-2507` is non-thinking-only (no `<think>`
   capability), smaller/faster, and out-ranks Qwen3-8B as a fine-tuning base.
4. **Eval too weak**: 10 LLM-graded scenarios, single-sampled, ±1.5 noise; the
   "9.0 vs 8.0" win was statistically meaningless. The eval never appended the ACTION
   directive, so it never exercised the command path where the leak lived.

## Workstreams (ordered by execution)

### WS-F — Base + template config (prereq)
- `sweep-config.json`: primary base `Qwen/Qwen3-4B-Instruct-2507`; insurance arm
  `meta-llama/Llama-3.1-8B-Instruct` (already license-wired) and/or
  `mistralai/Ministral-3-8B-Instruct-2512`. All non-thinking instruct.
- Lift `transformers` pin (`requirements.txt` already `>=4.51,<5` on the Vertex path;
  confirm), keep `max_seq 8192`.
- Remove the `<think>` scaffolding from BOTH `train.py` CHATML and
  `Modelfile.qwen-v2.tmpl` (non-thinking base → no think machinery, train==serve).

### WS-1 — Assistant-only loss (highest leverage)
- `train.py`: wrap the assistant turn in the CHATML with `{% generation %}…{% endgeneration %}`
  and set `SFTConfig(assistant_only_loss=True)`. Verify the mask fires on one batch
  (`return_assistant_tokens_mask`) before a full run — TRL's auto-patch won't fire on
  an overridden template string, so the markers must be manual.
- Demote `val_loss` to a sanity floor; WARN if `< 0.05` (memorization signal).

### WS-3 — Structural JSON guard (no retrain; also fixes current DeepSeek)
- `lib/extractAction.ts`: strip EVERY balanced top-level JSON object (and all fenced
  regions), not just the first; keep the first `action`-bearing object as payload.
  TDD against the exact v2 double-emission string + existing `test-extract-action.mjs`.
- `app/api/kitchen-assistant/route.js`: sanitize `finalAnswer` so no raw `{"action"`
  or ```` ```json ```` can reach the rendered answer (belt over the extractAction fix).
- New Node regression test: stub `ollamaChat` to return known-bad strings (double
  JSON, `<think>` block, in-prose numbers) and assert the rendered `answer` is clean.
- (Optional, assess) Ollama `format`=action-union-schema on the command path
  (`isCommand`) for a decoding-level guarantee; server templates the confirmation.

### WS-6 — Serving speed
- `route.js`: reorder `userContent` so the byte-stable static prefix (GROUNDED_SYSTEM
  as system msg + catalogs + directive) leads and the dynamic CONTEXT + history + cook
  message trail — unlocks Ollama's prefix KV cache. Re-run eval to confirm behavior
  unchanged.
- Config pack (docs + `Modelfile.qwen-v2.tmpl`): `OLLAMA_FLASH_ATTENTION=1`,
  `OLLAMA_KV_CACHE_TYPE=q8_0`, `num_ctx 8192` (prompt is ~5–6k tok), long keep-alive,
  cap `num_predict`.

### WS-4 — Eval overhaul (trustworthy gate)
- New deterministic format linter (`training/eval/format-lint.mjs`, TDD): on command
  prompts assert exactly one schema-valid action, no 2nd JSON, no `<think>`, no numeric
  quantities in trailing prose; on question prompts assert no action JSON except
  db_query/semantic_search, allergen answers never say "safe/free of", HACCP answers
  cite the exact number. Hard pre-gate before any LLM grading.
- `run-eval.mjs`: rebuild eval prompts from the production `buildRuntimeUserMessage`
  so command scenarios carry the ACTION directive (byte-matched to serving).
- `evaluate-candidates.mjs`: N=5 paired multi-sample per scenario, bootstrap 95% CI on
  candidate−incumbent margin; FLIP only if lower CI bound > 0 AND mean margin ≥ 1.0.
- Real command-path latency gate (full ~15k-char prompt, num_predict 512): p95 < 30s.
- Expand scenarios (→ ~40–60) + a frozen held-out TEST split; add adversarial traps
  (double-JSON bait, `<think>` bait, allergen "is it safe?" traps).

### WS-5 — Data upgrade
- `generate.mjs`: hard-filter any target with a 2nd action-object, a `<think>` block,
  or "safe/free of/does not contain" (outside an explicit escalation). New invariants.
- Build a held-out gold TEST set (~120–200 items) in the runtime prompt shape,
  contexts held out of train.
- (Assess) teacher-distill the *prose* targets over real contexts with
  `Qwen3-30B-A3B-Instruct-2507` (Apache-2.0) or the DeepSeek baseline as co-teacher,
  judge-filtered; keep the action-JSON payloads deterministic.

### WS-2 — Re-sweep, gated eval, flip
- Regenerate dataset (WS-1/WS-5), re-upload, launch multi-arm sweep (4B + 8B).
- Download, run the new gate (WS-4), flip only if a candidate beats DeepSeek beyond
  noise; else stay on DeepSeek and report honestly.
- Verify gates, GUI smoke (now part of the gate), PR, teardown.

## Non-negotiables
- Every code change TDD'd; `npm run verify` green before commit.
- The structural JSON guard (WS-3) lands even if no model is flipped — it fixes the
  leak for the current DeepSeek deployment too.
- WS-3/WS-6 touch the production web-app route; the PR must call that out.
- No flip unless the gate shows a margin exceeding grader noise.
