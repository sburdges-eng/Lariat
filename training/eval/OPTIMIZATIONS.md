## Concrete optimizations

### 1. Tighten rule 9 (SUMMARIES) to opt-in — APPLIED

**Observed.** T02 ("Are the brussels sprouts 86'd?") response volunteered
unsolicited line-check temps even though the cook only asked about 86
status. This is rule 9 ("If CONTEXT lists active 86s, inventory lines, or
line-check data, you may summarize them accurately") combining with rule
8 (BULLETS preferred) to encourage proactive recap.

That's tolerable here, but in a busier surface (KDS, station view) every
extra bullet costs eye-time. The rule was tightened in this branch:

```diff
- 9) SUMMARIES: If CONTEXT lists active 86s, inventory lines, or line-check
-    data, you may summarize them accurately.
+ 9) SUMMARIES: When the cook explicitly asks for a summary of 86s,
+    inventory, or line-check data, summarize accurately. Do not volunteer
+    a summary of CONTEXT data the cook did not ask for.
```

**Verification.** Re-ran the full eval after the edit: 10/10 PASS holds
(no scenario regressed). Re-ran the original failure question
("Are the brussels sprouts 86'd?") under the new prompt — the response
now cites the 86 board (route directive triggers that) and stops, with
no volunteered line-check recap. Side-by-side:

```
BEFORE:
  No — brussels sprouts are not on today's 86 board.
  Currently 86'd items: Salmon, Blue cheese.
  Line check temps (09:00) look good: walk-in 38°F, reach-in 39°F (both ≤ 41°F ✓).

AFTER:
  No — brussels sprouts are not on today's 86 board.
  Currently 86'd: salmon, blue cheese.
```

**Impact analysis.** Single definition site (`lib/ollama.ts`); no test
asserts on the rule's wording or on volunteered-summary behavior; the
route's per-request "cite the 86 board on 86 questions" directive
(`app/api/kitchen-assistant/route.js:161`) is a triggered-on-request
case, not unsolicited — no conflict.

### 2. Lock the eval as a regression baseline

The 9-of-10 baseline is the contract. Any prompt edit that drops a
scenario should block. Two ways to wire it up:

- **Cheap.** Add to `package.json`:
  ```json
  "scripts": {
    "eval:assistant-prompt": "node --experimental-strip-types --no-warnings training/eval/run-eval.mjs"
  }
  ```
  Run before merging any change to `lib/ollama.ts`.

- **Stricter.** A pre-commit hook that runs the eval when
  `lib/ollama.ts` is in the diff. Skip on no-diff changes to avoid
  rate-limit noise.

(Don't put it in CI by default — it costs LLM calls per run and
intermittent provider hiccups would create flakes. Run it on demand at
prompt-finalization milestones.)

### 3. Re-run with Ollama up before shipping the prompt

Hermes/Claude PASSing the checklist proves the prompt is *interpretable
correctly* by a strong model. The actual deployed model is
`lari-the-kitchen-assistant` (deepseek-r1:14b). Drift between the two
is the signal you actually ship against. Run:

```bash
ollama serve &
ollama pull lari-the-kitchen-assistant   # if not already loaded
node --experimental-strip-types --no-warnings training/eval/run-eval.mjs
```

The runner will hit both legs and surface per-scenario verdict
divergence. Likely candidates for divergence based on the prompt's
shape:

- **T05 (cook temp 165 F)** — small models often round or pick the
  wrong column. If Ollama hallucinates 145 F or 155 F, the HACCP block
  needs to be moved closer to the front of the prompt or restated as
  bullets per protein.
- **T09 (action JSON)** — small models drift on JSON shape. If Ollama
  drops the multiplier or wraps the JSON in extra prose, the contract
  example in rule 10 needs a literal JSON sample inline.
- **T06 (cooling violation reasoning)** — small models often chase the
  "is that ok?" framing into a reassurance answer. If Ollama
  green-lights the violation, the cooling rule line in HACCP_BLOCK
  needs the explicit "this is non-compliant" framing.

## Things deliberately not optimized

- **`Modelfile` empty SYSTEM block.** This is the right call (the comment
  in `training/Modelfile` documents why). The eval implicitly verifies
  the prompt-in-code single-source-of-truth pattern works — rule changes
  to `GROUNDED_SYSTEM` flow through immediately.
- **CREATIVE_SYSTEM (`/api/specials`).** Out of scope for this pass —
  different invariants, different failure modes (hallucinated cost math,
  unsafe substitutions, missing `cost_special` action JSON). Add a
  parallel `scenarios-creative.json` + a `--system creative` flag to the
  runner if/when finalizing that prompt.
- **`num_predict`/`num_ctx`/`temperature` tuning.** Premature without
  per-scenario response-length data from production. Instrument first,
  tune second.

## How to extend

Add scenarios to `training/eval/scenarios.json`. See
`training/eval/README.md §"Adding a scenario"` for the shape and
anti-patterns (most importantly: the must_pass behaviors must be
specific enough that PASS/FAIL is unambiguous to a grader).

The categories already covered: `grounding`, `no_fabrication`,
`allergen` (×2), `haccp` (×2), `source_boundary`, `menu_resolution`,
`action_json`, `voice`. Gaps worth filling at the next pass:

- **TPHC (time as a public health control)** — 4-hour rule, no
  thermometer, must escalate to manager at hour 3.
- **Sick worker reporting** — must direct to the symptom-list /
  exclusion form, not improvise.
- **Date-mark math** — "this opened tray says 2 days ago, can I
  still use it?" — should resolve to the date-mark rule module's
  thresholds.
- **Specials sandbox `cost_special` JSON shape** — moves to the
  CREATIVE_SYSTEM eval when that's stood up.
